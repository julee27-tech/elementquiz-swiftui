01 · The mental model shift
UIKit / Storyboard
Imperative — tell the UI what to do
ViewController owns everything
Outlets + IBActions wired in canvas
You manually sync data → UI
Class-based, reference semantics
→
SwiftUI
Declarative — describe what it should look like
ViewModel owns logic, View owns layout
@State, @Binding, closures inline
UI automatically reacts to state changes
Struct-based, value semantics
02 · Core SwiftUI concepts
@State
Local view state
Marks a value owned by a view. When it changes, SwiftUI automatically re-renders the relevant parts — no updateElement() needed.
Was → var currentElementIndex = 0
@Binding
Two-way connection
Lets a child view read AND write a value that lives in a parent. Pass with $ prefix. No delegate pattern, no callbacks.
Was → @IBAction + @IBOutlet pair
@ObservableObject
Shared reactive state
A class whose @Published properties trigger view updates across the whole app. The ViewModel uses this to broadcast changes.
Was → Shared data model + manual reload
View struct
The new ViewController
Every UI component is a struct conforming to View. No viewDidLoad, no outlets — just a body property returning the layout.
Was → class ViewController: UIViewController
.modifier()
Chained configuration
Configure views by chaining modifiers. Each returns a new view with that change applied. Order matters — think of it as a pipeline.
Was → label.font = ...; label.textColor = ...
body: some View
Declarative UI description
body is a computed property. SwiftUI calls it whenever state changes. You return a description of the UI — SwiftUI handles the diff and update.
Was → viewDidLoad() + manual setup calls
03 · Side-by-side code comparison
Displaying the current element
UIKit
func updateElement() {
  let name = elementList[index]
  imageView.image =
    UIImage(named: name)  // imperative
  answerLabel.text = "?"
  // Must call manually every time
}
SwiftUI
Image(vm.currentElement.name)
  .resizable()
  .scaledToFit()  // declarative

// Automatically updates when
// vm.currentIndex changes
Showing the answer
UIKit
// Requires @IBOutlet + @IBAction
// wired in Interface Builder
@IBAction func showAnswer(_ s: Any) {
  answerLabel.text =
    elementList[currentIndex]
}
SwiftUI
Text(vm.isRevealed
  ? vm.currentElement.name
  : "?")

Button("Show Answer") {
  vm.isRevealed = true  // one line
}
Navigating to the next element
UIKit
@IBAction func next(_ s: Any) {
  currentIndex += 1
  if currentIndex >= count {
    currentIndex = 0
  }
  updateElement()  // easy to forget
}
SwiftUI
// In ViewModel:
func nextElement() {
  currentIndex =
    (currentIndex + 1) % count
  isRevealed = false  // auto-reset
}
// Button calls vm.nextElement()

05 · Golden rules for junior devs
01
State lives in one place
Never duplicate state across two views. One source of truth, passed down via @Binding or @ObservedObject.
02
Views describe, don't act
A view should return layout and read state. Logic (next, reveal) goes in the ViewModel, not in body.
03
$ means two-way
Pass $value when a child needs to write back. Pass value (no $) when the child only reads it.
04
Small views, clear names
If body gets long, extract a subview. ElementCardView is easier to find and edit than 40 lines buried in ContentView.
05
@StateObject owns, @ObservedObject borrows
The view that creates the ViewModel uses @StateObject. Views that receive it from a parent use @ObservedObject.
06
Forget updateUI()
You will never call a refresh method. Mutate @Published state — SwiftUI handles the rest automatically.
